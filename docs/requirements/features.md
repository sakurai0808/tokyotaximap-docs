# 機能要件

## 検索機能

運行中の第一導線は検索にする。検索ワードに関するコンテンツを一覧できるようにするため、本文全文を検索対象にする。運行時に初めてのビルを調べる場合は、検索を行わせることで目的の記事に素早く辿り着けるようにする。

タクシーでは「過去に使われていた名称」や「正式名称が一般的には定義されておらず、ユーザーによって呼び方がまちまち」なものが多くある。記事 front matter の `keywords` としてそれらの名称も登録しておくことで検索効率を上昇させる。

サジェスト機能を導入し、検索ボックスに文字を入力している間に「（自分が知りたいのは）これかも」と思わせられるようにする。`title` と `keywords` の両方でマッチさせ、候補の表示文言は常に `title` とする。候補をタップすると検索ボックスに候補のタイトルが挿入され、すぐに検索ボタンの押下につなげられるようにする。

固定ヘッダーの検索ボックスで検索ボタンを押したあとの記事一覧ページは、サジェストから候補を入れたあとも必ず挟む。これは「目的の記事に素早く辿り着ける」目的には貢献しない。しかしブログのようなコンテンツサイトとしては一般的な挙動であり、運行中のような時間がタイトなときでもストレスをかける要因にはならないと考えているからである。

## ピンマップ

サイト内のコンテンツを地図内で一覧できる。目的のピンを選択するとポップアップが出現し、概要文、Googleマップへのリンク、当サイトの解説記事へのリンクが表示される。解説記事のリンクからは記事へ直行する。

参考: [タクシー運転手向けピンマップ（テスト）](https://sample-map-eight.vercel.app/)

掲載対象は記事 front matter の `pinmap: true` とする。カテゴリごとの目安は次のとおり。

- 交差点 — 掲載する
- 主要駅 — 掲載する
- 施設 — 掲載する
- 抜け道・定番ルート — 抜け道のみ掲載（`pinmap: true`）。定番ルートは指し示す範囲が広いため `pinmap: false`
- 通称がある道 — 掲載する
- 首都高 — 出入口を掲載する

幹線道路、タクシー基礎などは運行中の使用を想定しない「読み物」であり、ピンとして表示する必要性が薄いため掲載しない。

ピンマップは第二の導線とする。検索は仕事に出たばかりの新人も対象とする一方、ピンマップは3か月程度の営業経験があるドライバー、またはそれ以降のドライバーを想定する。

例えば「首都高の入口から入りたいが、方向によって右折禁止がなかったか」をすぐ確認したい場面で使用する。休憩時の俯瞰を主目的とはしないが、そのような利用を妨げるものではない。

## YouTube

自社チャンネルを運営する。現在の本数は30本ほど。幹線道路、施設の進入路、抜け道など、ビジュアルで覚えるのが効果的な記事に紐づけて動画を作成する。

基本的に記事ページへ動画を埋め込み、画面遷移のわずらわしさを軽減する。「無線連絡で初めて行くビルを指定された」といった停車中の場面でも、短時間で確認できるものを想定する。

ただし「幹線道路」や「交差点」カテゴリの動画などは、ゆっくり流し見することを想定し、この場面での視聴対象とはしない。

## QRコードによる共有

記事ごとにQRコードを表示し、運転手と乗客が同じURLを共有できるようにする。運転手、乗客ともにスマートフォンの利用を想定し、各社独自の車載ナビは対象にしない。

### 実際のシチュエーション

1. 乗客が乗車する。
2. 乗客が知名度のある行き先を伝えるが、運転手が分からない。
3. 停車した状態で、お互いにスマートフォンで調べる。
4. 乗客が当サイトのページを発見し、「QRコードで共有」ボタンを押して運転手に見せる。
5. 運転手がスマートフォンのカメラで読み取り、該当ページから情報を得る。
