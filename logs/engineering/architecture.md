## まず前提の確認

## 選択肢は実質 3 つ

WordPress の経験があると「CMS か否か」の二択に見えますが、Markdown 前提だと実際には次の 3 層になります。

**A. Git 管理（リポジトリ内に `.md` ファイル + front matter）**
記事はリポジトリの中のファイル。メタデータは front matter（YAML）で持つ。

```markdown
---
title: 東京ミッドタウン
category: 施設
wards: [港区]
keywords: [ミッドタウン, ミッドタウンビル]
pinmap: true
location: { lat: 35.6656, lng: 139.7312 }
youtube: https://youtu.be/xxxx
---

本文...
```

**B. Git ベースのヘッドレス CMS（Decap CMS / TinaCMS など）**
実体は Git 上の `.md`（=A と同じ）。その上に**編集用ダッシュボード UI**をかぶせる。プレビューやフォーム入力ができるが、保存先はリポジトリ。

**C. API ベースのヘッドレス CMS（microCMS / Contentful など）**
記事はクラウドの DB に入り、ビルド時/実行時に API で取得。Markdown で書けるものもあるが、実体は Git 外。

## あなたの直感は、この規模では正しい

「エンジニアが管理するだけなら Git のほうが AI にも読ませやすく、システムとして管理しやすい」というのは、**この案件では的を射ています**。理由:

| 観点                     | Git 管理 (A)                        | API ヘッドレス CMS (C)              |
| ------------------------ | ----------------------------------- | ----------------------------------- |
| AI への読み込み          | ◎ ファイルをそのまま渡せる          | △ API 経由で吸い出す一手間          |
| URL 引き継ぎ（移行要件） | ◎ ファイル名/slug で制御しやすい    | ○ 設定次第                          |
| 全文検索インデックス構築 | ◎ ビルド時に全 `.md` を走査するだけ | △ 一度全件取得が必要                |
| バージョン管理・差分     | ◎ Git そのもの                      | △ CMS 側の履歴機能に依存            |
| 費用                     | ◎ 無料（リポジトリのみ）            | △ 従量/月額、600 本規模で効いてくる |
| ロックイン               | ◎ ただのファイル                    | △ 製品依存                          |
| リライト運用（順次修正） | ◎ 差分で追いやすい                  | ○                                   |

特に、この案件の肝である **全文検索+サジェスト** と **AI によるリライト支援** は、記事がプレーンな `.md` としてリポジトリにある状態が圧倒的に相性が良いです。ビルド時に全ファイルを読んで検索インデックス（例: FlexSearch/Pagefind 等）を生成する、という素直な構成が組めます。

## CMS の利点（ダッシュボード）は、こう埋められる

WordPress で感じていた「わかりやすいダッシュボード」の価値は、主に次の 3 つに分解できます。そしてそれぞれ Git 運用でも代替可能です。

1. **フォーム入力でメタ data 入力ミスを防ぐ** → front matter のスキーマを Zod 等で検証、または後述の B を導入
2. **プレビュー** → Next.js の dev/preview 機能で十分
3. **非エンジニアでも書ける** → ここだけが Git の弱点

## 検索との関係

要件を整理すると、検索インデックスに載せる情報源は 3 つあります。

| 情報源           | どこから来るか                  | 役割                               |
| ---------------- | ------------------------------- | ---------------------------------- |
| 本文全文         | Markdown 本文                   | 記事内の説明を横断検索             |
| 別名・キーワード | **front matter**                | タイトルにない呼び方・旧名称を拾う |
| ミニ単語帳       | 別ファイル（全区共通 1 ページ） | 用語検索 → 単語帳の該当箇所へ      |

front matter が検索で特に効くのは **`aliases`（別名）** あたりです。`features.md` にある「正式名称が定まっていない」「呼び方がまちまち」なケースは、本文に書いていてもヒットしにくいことがあります。意図的に登録しておく価値があります。

```7:7:docs/requirements/features.md
タクシーでは「過去に使われていた名称」や「正式名称が一般的には定義されておらず、ユーザーによって呼び方がまちまち」なものが多くある。キーワードとしてそれらの名称も登録しておくことで検索効率を上昇させる。
```

一方 **サジェスト** は要件上、候補タップで **タイトル** が検索ボックスに入ります。なのでサジェストの主データは `title` で、`aliases` は「検索実行時のヒット」向け、という分担が自然です。

```9:9:docs/requirements/features.md
サジェスト機能を導入し、検索ボックスに文字を入力している間に「（自分が知りたいのは）これかも」と思わせられるようにする。候補をタップすると検索ボックスに候補のタイトルが挿入され、すぐに検索ボタンの押下につなげられるようにする。
```

## front matter の設計方針

キーを「検索用」「表示・導線用」「地図用」に分けて考えると整理しやすいです。

### 1. 全記事共通（必須）

```yaml
---
title: 東京ミッドタウン # 表示名・サジェスト・検索（高ウェイト）
slug: tokyo-midtown # URL引き継ぎ（移行要件）
category: 施設 # カテゴリ一覧・さくいん
wards: [港区] # 区絞り込み（複数可、空配列も可）
aliases: # 検索専用の別名（本文に無くてもよい）
  - ミッドタウン
  - ミッドタウンビル
thumbnail: /images/articles/tokyo-midtown.jpg
---
```

- **`slug`** は `content-model.md` にはないですが、`migration.md` の URL 引き継ぎのために実質必須です
- **`wards`** は `[]` を許容（タクシー基礎など区なし記事）
- **`aliases`** は配列 1 本にまとめるのがおすすめ。「別名」と「キーワード」を分けると、どちらに何を書くか迷いが生じます

### 2. ピンマップ対象のみ（条件付き）

```yaml
pinmap: true
location:
  lat: 35.6656
  lng: 139.7312
summary: 六本木ヒルズ隣。赤坂方面からは… # ポップアップ用の短い概要
```

ピンマップ対象カテゴリは要件で決まっています（交差点・主要駅・施設・抜け道・知名度の高い道・首都高）。`pinmap: true/false` を明示しておくと、カテゴリから自動判定するより例外（座標未確定など）を扱いやすいです。

### 3. 任意

```yaml
youtube: https://www.youtube.com/watch?v=xxxx
publishedAt: 2026-01-15 # 将来の並び替え用（今は必須にしなくてよい）
updatedAt: 2026-07-01
```
